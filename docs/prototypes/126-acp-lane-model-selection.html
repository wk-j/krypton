A · Lane rail — model chip statesships v1

The model chip is the only visible model surface in v1. It shows the configured model (intent), not a confirmed-active id (session/set_model returns {}). On apply failure the same chip text is kept but flagged with a warning style + tooltip — it never guesses a real model. Backends that don't advertise model state show no chip.

B · How you configure it — krypton.tomlships v1

v1 is config-driven (no in-app editor). The existing lane_models.<backend> table now actually takes effect for model-advertising adapters (Claude verified). Keyed by backend id, so two concurrent claude lanes share one model in v1. 

# ~/.config/krypton/krypton.toml
[acp_harness.lane_models.claude]
active = "opus"          # the ONLY thing you set — alias or full id; adapter resolves "opus"→claude-opus-4-*
# models = [...]   ← OPTIONAL. You do NOT maintain a catalog: available models come from the
#                    agent (session/new). This array only curates/orders the future picker.

[acp_harness.lane_models.gemini]
active = "gemini-2.0-flash"   # applied as --model at spawn (unchanged path)

[acp_harness.lane_models.opencode]
active = "zai-coding-plan/glm-5.1"  # dedicated set_config_option path (unchanged)
Respawn-to-apply. Editing the TOML updates the chip/config but does not hot-swap a live lane's model. The agent model changes only on the next spawn — use #new, #new!, or restart the lane. (Matches the Gemini/Droid "change model ⇒ respawn" rule and the docs/83 "lane respawn is the trigger" precedent.)

C · Interactive model pickerfuture — deferred per spec 126

Not in v1. Sketched here to show the end-state "configure" UX and to inform whether to pull it forward. Trigger: leader → , (i.e. ⌘P then ,) — a new harness LeaderKeyBinding joining Add Lane +, Metrics =, Resume 0, Directives .. It can't be a plain letter (all letters are reserved global leader keys) nor ⌃/⌘M (memory drawer); , is free and mnemonic (the settings convention). A keyboard-driven overlay populated automatically from the agent-advertised availableModels (session/new response) — the user maintains no list; an optional lane_models.<backend>.models array only curates/orders this live set. Selecting issues a live session/set_model — which is why the follow-up must handle the adapter's mode/effort/context resync & visible mode downgrades that mid-session switching triggers.
